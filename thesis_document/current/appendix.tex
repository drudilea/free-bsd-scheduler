\chapter{Detalles de la implementación del módulo de monopolizado}\label{appendix:apB}

En este apartado se desarrollará con mayor profundidad la implementación del módulo de monopolización de hilos, haciendo referencia principalmente a las funciones que se han modificado y su comportamiento a lo largo del proceso de ejecución de un hilo.

El planificador 4BSD maneja los cambios de contexto cuando un hilo termina su ejecución o queda bloqueado, generando una interrupción para seleccionar un nuevo hilo.

La función principal para gestionar estos cambios es \verb|mi_switch()|, que invoca a \verb|sched_switch()|. Esta función se encarga de preparar el nuevo hilo para su ejecución y usa \verb|sched_add()| para añadirlo a la cola de un procesador, considerando la afinidad y las políticas del planificador. Luego, \verb|sched_choose()| selecciona el hilo a ejecutar y realiza el cambio de contexto en caso de ser necesario.

Para integrar el módulo de monopolización, modificamos \verb|sched_add()| para permitir que ciertos hilos se fijen a CPUs específicas. Esto asegura que una CPU ejecute siempre el mismo hilo hasta que se decida lo contrario, excluyendo así, a todos aquellos hilos con un identificador diferente al elegido.

Comenzamos implementando un arreglo llamado \verb|pinned_threads_per_cpu|, que almacena el identificador (ID) del hilo asignado a cada CPU. El arreglo contiene tantos elementos como CPUs existan en el sistema, es decir, para un procesador de cuatro núcleos, el arreglo contendrá cuatro elementos. Cada elemento será el encargado de guardar el valor del identificador del hilo que monopoliza la CPU correspondiente. Si una CPU está libre, el valor en el arreglo es -1.

En el siguiente ejemplo, se detalla un caso en el que el hilo con ID 100101 tomó control sobre la CPU1 y donde el resto de los procesadores se encuentran funcionando normalmente.

int \verb|pinned_threads_per_cpu|[CPU\_NUMBER] = \{ -1, 100101, -1, -1 \};

Al momento de encolar el hilo, se busca cuál de los procesadores disponibles sería la mejor opción para continuar la ejecución del mismo. Esta decisión se toma dentro del método \verb|resource_choose_cpu()| desarrollado en el trabajo integrador previo y extendido actualmente, para hacer uso de este nuevo arreglo de hilos asociados a procesadores. El método recibe como parámetro el hilo que se quiera encolar, y realiza tres operaciones condicionales que determinarán el procesador a elegir:

\begin{itemize}
    \item Como primera condición, si el hilo se encuentra dentro del arreglo de \verb|pinned_threads_per_cpu| ya estamos en condiciones de elegir dicho procesador como el indicado para el encolado.
    \item Si el hilo no se encuentra dentro del arreglo, se intenta asignar a la cola del último procesador en el que se ejecutó. Esto es posible solo si la transición \verb|TRAN_ADDTOQUEUE| de dicho procesador se encuentra sensibilizada y el procesador no está monopolizado por otro hilo.
    \item Por último, si no se cumplen ninguna de las dos condiciones previas, se recorren los diferentes núcleos del procesador y se retorna el primero que cumpla las condiciones necesarias para el encolado.
\end{itemize}

Como parte del desarrollo del módulo, también se implementaron algunos métodos complementarios encargados de manejar el estado del arreglo \verb|pinned_threads_per_cpu|.

\begin{itemize}
    \item \verb|toggle_pin_thread_to_cpu()|: Método encargado de conmutar el estado de monopolización de un procesador. Recibe el ID de un hilo y de un procesador como parámetros y realiza diferentes operaciones de acuerdo al estado del arreglo:
          \begin{itemize}
              \item Si el procesador se encuentra libre, se agrega el ID del hilo a la posición correspondiente en el arreglo.
              \item Si el procesador ya estaba monopolizado por otro hilo, se sobrescribe con el nuevo ID.
              \item En caso de que el hilo ya se encuentre monopolizando al procesador, lo libera escribiendo el valor -1 en la posición correspondiente.
          \end{itemize}
    \item \verb|cpu_available_for_thread()|: Método utilizado por \verb|resource_choose_cpu()| para saber si un hilo puede utilizar un procesador, o si este se encuentra monopolizado por otro. Recibe el ID de un hilo y de un procesador como parámetros y retorna 1 en caso de que el procesador se encuentre habilitado para encolar dicho hilo; retorna 0 en caso de que dicho procesador se encuentre tomado por otro hilo.
    \item \verb|get_monopolized_cpu_by_thread_id()|: Método utilizado por \verb|resource_choose_cpu()| para obtener el ID del procesador al que se encuentra asociado el hilo enviado por parámetro. Retorna -1 en caso de que no esté anclado a ningún CPU.\par
\end{itemize}

\section{Interpretación posterior del fallo}

El análisis inicial permitió identificar el lugar en el que el \textit{kernel} terminaba fallando, pero no explicaba por sí solo la causa del problema. La aparición del fallo en \verb|kern_select()| indicaba el punto en el que se observaba el acceso inválido a memoria, aunque el error se originaba antes, durante una decisión de encolado del planificador.\par

La causa identificada fue una desincronización entre las run queues reales de FreeBSD y el marcado de la red de Petri. En las versiones previas del planificador, podía ocurrir que FreeBSD encolara un hilo en una cola per-CPU por una restricción de \textit{pinning}, \textit{bounding} o afinidad, mientras que la red no registraba el disparo de \verb|TRAN_ADDTOQUEUE| porque dicha transición se encontraba inhibida por \verb|PLACE_CANTQ|. Desde ese momento, la red dejaba de representar correctamente el estado del planificador.\par

El efecto posterior era que, cuando el \textit{scheduler} intentaba desencolar o ejecutar el hilo, la red podía intentar disparar transiciones como \verb|TRAN_UNQUEUE|, \verb|TRAN_EXEC| o \verb|TRAN_REMOVE_QUEUE| sobre un marcado incorrecto. Por eso el \textit{panic} aparecía en funciones alejadas del punto original del error: el \textit{crash} era una consecuencia tardía de una decisión de encolado que había dejado inconsistentes al \textit{kernel} y al modelo.\par

Para corregir esta situación se incorporó la transición \verb|TRAN_ADDTOQUEUE_FORCED|. Esta transición registra en la red los encolados que FreeBSD debe realizar obligatoriamente para respetar reglas internas del sistema operativo. A diferencia de \verb|TRAN_ADDTOQUEUE|, no está inhibida por \verb|PLACE_CANTQ|, pero mantiene la incidencia sobre \verb|PLACE_QUEUE| y \verb|PLACE_CANTQ|. De esta forma, evita la desincronización sin eliminar la penalización de la CPU para futuras decisiones de balanceo.\par

\section{Instrumentación dinámica con DTrace}
\label{appendix:dtrace-sdt}

\textit{DTrace} es un framework de observabilidad dinámica que permite inspeccionar eventos del sistema durante la ejecución, activando trazas bajo demanda. A diferencia de una depuración basada en impresiones permanentes en el \textit{kernel}, permite recolectar información puntual durante una prueba y luego desactivar la instrumentación sin mantener una salida constante de logs.\par

Un \textit{probe} es un punto observable sobre el cual \textit{DTrace} puede enganchar una acción de trazado. Algunos probes son provistos por el sistema, mientras que otros pueden ser definidos explícitamente dentro del código. Para este trabajo se utilizó \verb|SDT| (\textit{Statically Defined Tracing}), que permite declarar probes estáticos propios en el \textit{kernel}. De esta forma se expusieron eventos internos del planificador y de la red de Petri, como decisiones tomadas por \verb|sched_add()|, transiciones efectivamente disparadas y transiciones que hubieran quedado bloqueadas por falta de sensibilización.\par

Esta instrumentación fue necesaria porque el problema no se manifestaba en el mismo punto en el que se originaba. El \textit{page fault} aparecía más tarde, en funciones alejadas del planificador, por lo que era necesario observar los eventos previos de encolado y el efecto que tenían sobre el marcado de la red. Con probes propios, las pruebas pudieron diferenciar si un hilo iba a la cola global, si era encolado por la política normal de Petri o si era registrado mediante el camino forzado agregado para respetar restricciones internas de FreeBSD.\par

No se registraron transiciones bloqueadas durante las pruebas, lo que indica que el modelo permaneció sincronizado con el estado real del planificador en los escenarios evaluados.\par
